01

Overview

The ShieldWatch Identity API is a RESTful API enabling partners, employers, and enterprise clients to integrate identity protection services directly into their platforms. Common integration patterns include employee benefit portals, HR platforms, financial institution dashboards, and white-labeled consumer identity products.

This playbook covers everything needed to go from zero to a fully operational integration — from obtaining credentials to processing real-time webhook alerts.

Base URLs

EnvironmentBase URLNotes
sandboxhttps://api-sandbox.shieldwatch.io/v2Full feature set, mock data, no billing
productionhttps://api.shieldwatch.io/v2Live data, SLA-backed

API Design Principles

  • RESTful — standard HTTP verbs (GET, POST, PATCH, DELETE) with resource-based URL paths
  • JSON — all request and response bodies are application/json
  • Versioned — major version in URL path (/v2/); non-breaking changes are additive
  • Idempotent writes — POST endpoints accept an Idempotency-Key header (UUID v4 recommended) to prevent duplicate operations; keys are honored for 24 hours
  • Pagination — list endpoints use cursor-based pagination via after / before parameters
Request format

Content-Type: application/json
Accept: application/json

Timestamps

All timestamps are ISO 8601 UTC strings, e.g. 2026-05-16T14:30:00Z

ShieldWatch supports four authentication methods to accommodate different integration patterns — from simple server-to-server API calls to full enterprise SSO deployments.

2.1 API Key Authentication

The simplest method. Suitable for server-to-server integrations where a human user session is not required. Pass your API key in the X-API-Key header.

✅
PKCE Required (OAuth 2.1)
All Authorization Code flows must use PKCE (RFC 7636) regardless of whether a client secret is present. This prevents authorization code interception attacks and aligns with OAuth 2.1 (draft-ietf-oauth-v2-1).
ℹ️
Redirect URI Exact Matching
Per OAuth 2.1, redirect URIs must match exactly — no partial path matching, wildcard subdomains, or localhost exceptions in production. Register each URI explicitly in the developer portal.

Widget Embedding & Token Exchange

The ShieldWatch widget can be embedded directly into a partner portal via <iframe>, surfacing real-time identity alerts without redirecting users away. The widget authenticates via a short-lived session token exchanged server-side.

4.1 Token Exchange Flow

Your backend exchanges its OAuth access token for a user-scoped widget token. The widget token is single-use and expires in 15 minutes — it cannot be used for API calls.

Partner Backend ──→ ShieldWatch API : POST /widget/token (with member_id + Bearer) ShieldWatch API ──→ Partner Backend : { widget_token, expires_in: 900 } Partner Backend ──→ Browser : renders page with widget_token in HTML Browser ──→ iframe : loads widget src URL with token param iframe ──→ ShieldWatch CDN : validates token, renders member dashboard

window.addEventListener('message', (event) => {
  // Always validate the origin
  if (event.origin !== 'https://widget.shieldwatch.io') return;

  const { type, payload } = event.data;

  if (type === 'SW_ALERT_CLICK') {
    // Open alert detail in your own UI
    openAlertPanel(payload.alert_id);
  }

  if (type === 'SW_SCORE_CHANGE') {
    updateScoreBadge(payload.new_score);
  }

6.4 Signature Verification

Every webhook request includes a SW-Signature header containing an HMAC-SHA256 signature. Always verify this before processing the payload.

Verify Signature (Node.js)JS
const crypto = require('crypto');

function verifyWebhook(rawBody, signatureHeader, secret) {
  const [version, receivedSig] = signatureHeader.split(',');
  const timestamp = version.replace('t=', '');

  // Replay attack protection — reject if > 5 minutes old
  const age = Date.now() / 1000 - parseInt(timestamp);
  if (age > 300) throw new Error('Webhook too old');

  const payload  = `${timestamp}.${rawBody}`;
  const expected = crypto
    .createHmac('sha256', secret)
    .update(payload)
    .digest('hex');

  const sig = receivedSig.replace('v1=', '');
  return crypto.timingSafeEqual(
    Buffer.from(expected, 'hex'),
    Buffer.from(sig,      'hex')
  );
}
⚠️
Retry Behavior
ShieldWatch retries failed webhook deliveries (non-2xx or timeout) up to 5 times with exponential backoff: 5s → 30s → 2m → 10m → 1h. After 5 failures the endpoint is marked inactive and an alert is sent to the account owner.

Security Considerations

🚨
PII Handling
The ShieldWatch API handles sensitive PII including SSNs, financial data, and health records. All integrations must comply with applicable regulations (FCRA, GLBA, HIPAA where applicable). Consult your legal team before storing or logging API responses.

Transport Security

  • All API traffic requires TLS 1.2 or higher — TLS 1.0/1.1 are rejected; TLS 1.3 is preferred and negotiated by default
  • Certificate pinning is available for mobile SDK integrations
  • HSTS is enforced with a max-age of 1 year and includeSubDomains

Credential Management

  • Rotate API keys every 90 days or immediately upon suspected compromise
  • Store secrets in a secrets manager (AWS Secrets Manager, HashiCorp Vault, Azure Key Vault) — never in code or environment variables in shared systems
  • Use separate API keys for sandbox and production environments
  • Scope tokens to the minimum required permissions

Widget Security

  • Widget tokens are single-use and expire in 15 minutes — never cache them
  • Always generate widget tokens server-side; never expose your API key or OAuth client secret to the browser
  • Apply sandbox attribute to all iframes; only add permissions you explicitly need
  • Set a Content Security Policy (frame-src) on your host page limiting iframes to https://widget.shieldwatch.io

Webhook Security

  • Always verify the SW-Signature header before processing payloads
  • Reject payloads older than 5 minutes to prevent replay attacks
  • Use crypto.timingSafeEqual (or equivalent) for signature comparison to prevent timing attacks
  • Respond with 2xx within 30 seconds — long processing should be queued asynchronously
